# API 명세서: 인천 유나이티드 응원가 백엔드 연동

- **문서 버전**: v1.6
- **작성일**: 2026-07-13 (v1.6 업데이트: 2026-07-19)
- **연관 문서**: FUNCTION.md (F-01~F-04, F-11), PRD.md (v1.1, 6장 데이터 구조), TECHSTACK.md (v1.1, 2.3.1 API Mocking)
- **목적**: 지금 `data/songs.json` + `lib/api/songs.ts`로 하는 데이터 **조회(Read)** 를, 훗날 실제 백엔드(Route Handler + DB, 또는 Supabase)로 옮길 때 화면 코드가 그대로 쓸 수 있는 **조회 계약**을 정의합니다.
- **적용 범위**: 지금 화면(F-01, F-02, F-04, F-11)이 실제로 쓰는 조회(Read) API. *(2026-07-19 정리: 관리자 CRUD(등록/수정/삭제)는 F-14 관리자 페이지가 미채택으로 삭제되며 함께 제거했습니다 — 현재 앱은 정적 JSON을 직접 읽어 쓰는 백엔드·관리자 API가 없습니다. PRD 4장.)*

---

## 0. 지금 단계와 이 문서의 관계

- 현재 화면은 `lib/api/songs.ts`의 `getSongs()` / `getSongById(id)`만 호출하며, 내부에서 `data/songs.json`을 즉시 읽어 반환합니다(네트워크 호출 없음).
- 이 문서는 **`lib/api/songs.ts` 내부 구현을 아래 조회 엔드포인트 호출로 교체할 때의 계약(contract)**입니다. 화면(컴포넌트) 코드는 수정할 필요가 없습니다. (TECHSTACK.md 2.3.1 원칙 그대로 유지)

## 1. 공통 규칙

### 1.1 Base URL
```
/api/songs
```
Next.js Route Handler(`app/api/songs/route.ts`, `app/api/songs/[id]/route.ts`) 기준. 별도 백엔드 서버로 분리하더라도 경로 구조는 동일하게 유지합니다.

### 1.2 인증
- 조회(F-01~F-04, F-11)는 **인증 없이 공개** 접근합니다. (PRD Non-Goals — 로그인 기능 없음)

### 1.3 공통 응답 포맷 (성공/실패)

**성공**
```json
{
  "success": true,
  "data": { }
}
```
- `data`는 엔드포인트에 따라 객체 하나 또는 배열입니다.

**실패**
```json
{
  "success": false,
  "error": {
    "code": "SONG_NOT_FOUND",
    "message": "해당 응원가를 찾을 수 없습니다"
  }
}
```
- `message`는 기능명세서에 정의된 문구(Feedback 컴포넌트에 그대로 표시 가능한 한국어)를 그대로 사용합니다.

### 1.4 HTTP 상태 코드

| 상황 | 상태 코드 |
|------|-----------|
| 조회 성공 | 200 OK |
| 존재하지 않는 곡 접근 | 404 Not Found |
| 서버 내부 오류 | 500 Internal Server Error |

### 1.5 에러 코드

| code | 상태 코드 | 의미 | 대응 문구 (기능명세서 기준) |
|------|-----------|------|--------------------------------|
| SONG_NOT_FOUND | 404 | 존재하지 않는 `id` | "해당 응원가를 찾을 수 없습니다" (F-02) |
| INTERNAL_ERROR | 500 | 서버/DB 오류 | "잠시 후 다시 시도해 주세요" |

## 2. 데이터 모델

PRD.md 6장 · 기능명세서 "공통 데이터 규칙"과 동일하며, 필드명은 임의로 바꾸지 않습니다.

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| id | number | O (서버 자동 생성) | 곡 고유 번호. 생성 요청 시 클라이언트가 보내지 않음 |
| title | string | O | 응원가 제목 |
| videos | Video[] | X (기본값 `[]`) | 공식/현장 등 영상 목록 (F-04) |
| lyrics | string | X (기본값 `""`) | 가사, 줄바꿈(`\n`) 포함 |
| category | string | X (기본값 `"팀 응원가"`) | 분류. 현재 사용 값: `"팀 응원가"` / `"선수 응원가"` — F-01 기본 정렬의 묶음 기준 |
| tags | string[] | X (기본값 `[]`) | 현재 사용 값: `"미사용"`(팀을 떠난 선수의 곡 — F-01에서 배지 + 맨 아래), `"대표곡"`(예약) |
| isFavorite | boolean | X (기본값 `false`) | 예약 필드(현재 화면 미사용) — rules.md 2장에 따라 데이터 구조는 유지 |
| source | Source \| null | X (기본값 `null`) | **가사** 출처 (PRD 9장). 없으면 화면에 표기하지 않음 |

**Video 객체**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| type | `"official"` \| `"live"` | O | 영상 종류 |
| label | string | O | 탭에 보일 이름 (예: "공식 영상") |
| youtubeId | string | O | 유튜브 영상 ID |

**Source 객체** *(v1.2 신설)*

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| name | string | O | 화면에 보일 출처 이름 (예: `"인천네이션"`) |
| url | string | O | 원문 URL — 표기는 이 주소로 링크됩니다 |

- `videos`의 출처가 아니라 **`lyrics`의 출처**입니다. 영상은 유튜브 임베드라 출처가 자명하지만
  (PRD 9장), 가사는 어디서 옮겨왔는지 밝혀야 합니다.
- 크롤러(`scraper/chants/`)가 수집 시점에 채웁니다. 손으로 등록한 곡은 `null`입니다.
- **JSON에서의 표현**: 값이 없을 때 `"source": null`을 보내는 대신 **필드를 생략**해도 됩니다.
  화면은 두 경우를 동일하게 "출처 없음"으로 다룹니다. *(2026-07-17 기준 `data/songs.json`은
  34곡 전부 `source`가 있어 두 표현 모두 실제로는 등장하지 않습니다. 계약은 그대로 둡니다 —
  앞으로 손으로 추가하는 곡이 출처 없이 들어올 수 있습니다.)*

## 3. 엔드포인트

### 3.1 GET /api/songs — 목록 조회
- **연관 기능**: F-01 (목록), F-11(검색)
- **쿼리 파라미터** (선택, 지금은 미구현 — 확장 시 추가)

| 파라미터 | 설명 |
|----------|------|
| q | 제목/가사 검색어 (F-11) |

- *(v1.3)* **F-11 검색은 구현됐지만 `q`는 여전히 미구현입니다** — 모순이 아닙니다. 정렬과 같은 이유로 **검색도 화면(F-01) 책임**입니다: 34곡짜리 JSON을 통째로 받아 브라우저에서 거르므로 서버에 검색어를 보낼 일이 없습니다. `q`는 **곡이 수백 개로 늘어 전량 전송이 부담될 때**를 위한 자리로 남겨 둡니다. 그 시점엔 FUNCTION.md F-11의 '검색 버튼 없음' 결정도 함께 재검토해야 합니다(응답을 기다려야 하므로).

- **응답 200**
```json
{
  "success": true,
  "data": [
    {
      "id": 1,
      "title": "나의 사랑 인천 FC",
      "videos": [
        { "type": "official", "label": "공식 영상", "youtubeId": "8n6qHWq2e-A" },
        { "type": "live", "label": "현장 영상", "youtubeId": "uE0DAk5jeI4" }
      ],
      "lyrics": "오~ 오오오오! (인천!)\n...",
      "category": "팀 응원가",
      "tags": [],
      "isFavorite": false,
      "source": { "name": "인천네이션", "url": "https://incheonation.kr/free/6452" }
    }
  ]
}
```
- 곡이 하나도 없으면 `data: []` (200) — 화면은 이때 "등록된 응원가가 없습니다"(F-01 빈 상태)를 표시합니다. **빈 목록은 오류가 아닙니다.**
- **정렬은 화면(F-01) 책임입니다.** 서버는 저장 순서 그대로 반환하고, **정렬은 전부 프론트엔드에서** 합니다. 이 계약은 v1.2에서도 그대로입니다 — 정렬 규칙이 바뀌어도 API는 바뀌지 않습니다. *(v1.1 — '대표곡 우선 정렬'은 판단 기준의 주관성 문제로 제외. FUNCTION.md v1.1 참고)*
- *(v1.2)* 화면의 기본 정렬은 **분류 순서**입니다 — `category`(팀/선수 응원가)와 `tags`의 `"미사용"`을 읽어 **팀 응원가 → 선수 응원가(현역) → 미사용** 순으로 묶고, 묶음 안에서 `title` 가나다순(한국어 로케일)으로 정렬합니다. 상세는 FUNCTION.md v1.3 F-01.
  - **서버는 이 정렬에 관여하지 않습니다.** 응답에 `category`·`tags`만 정확히 담으면 됩니다.
  - ⚠️ **`title` 정렬 시 코드포인트 비교를 쓰지 마세요.** 한국어 로케일 비교(`localeCompare(title, "ko")` 등)를 써야 한글이 영어보다 앞섭니다. 코드포인트로 정렬하면 `"Bandiera"`가 `"회상"`보다 앞서 목록이 부자연스러워집니다(실제로 크롤러가 이 실수를 했습니다 — CRAWLER_SPEC.md 3.7.2).

### 3.2 GET /api/songs/{id} — 상세 조회
- **연관 기능**: F-02(상세), F-04(영상 탭은 응답의 `videos` 배열을 그대로 사용)
- **응답 200**: 2.의 데이터 모델 전체를 포함한 단일 객체 (`data`)
- **응답 404**
```json
{
  "success": false,
  "error": { "code": "SONG_NOT_FOUND", "message": "해당 응원가를 찾을 수 없습니다" }
}
```

> **생성/수정/삭제(CRUD) 엔드포인트는 정의하지 않습니다** *(2026-07-19 정리)*. 이를 쓰던 F-14(관리자 페이지)가 미채택으로 삭제됐고, 현재 앱은 응원가를 `data/songs.json`에 직접(또는 크롤러로) 넣습니다. 관리자 기능을 되살리면 그때 계약을 새로 정의합니다.

## 5. `lib/api/songs.ts` 연동 방향 (참고)

지금(Mock 단계)과 실제 API 연동 단계의 내부 구현만 바뀌고, 화면이 호출하는 함수 시그니처는 그대로 유지됩니다.

| 함수 | 지금 (Mock) | 연동 후 |
|------|-------------|---------|
| `getSongs()` | `songs.json` 배열 반환 | `GET /api/songs` 호출 → `data` 반환, 실패 시 예외 처리 |
| `getSongById(id)` | `songs.json`에서 `find` | `GET /api/songs/{id}` 호출 → 404면 `undefined` 반환(기존 계약 유지) |

이 표는 향후 실제 구현 작업 시 참고용입니다.

> **2장의 타입은 `lib/api/songs.schema.ts`에 있습니다** *(v1.3)* — `songs.ts`가 아닙니다. `songs.ts`는 `data/songs.json`을 import하므로, 화면이 거기서 **값**(상수·함수)을 가져가면 JSON 전체가 클라이언트 번들로 딸려 옵니다. 그래서 계약(타입·상수)만 따로 뒀습니다. 실제 API로 전환할 때도 이 분리는 유지하세요 — 바뀌는 건 `songs.ts` 내부뿐입니다. (TECHSTACK.md 2.3.2)
