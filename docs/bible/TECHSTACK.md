# 기술 스택 정의서: 인천 유나이티드 응원가 웹페이지

- **문서 버전**: v1.1
- **작성일**: 2026-07-11 (v1.1 업데이트)
- **연관 문서**: PRD.md (v1.1), 기능명세서.md (v1.0)
- **목적**: 이 웹을 어떤 기술·도구로 만들지 확정하고, 각 선택의 이유와 기능별 연결 관계를 정리하여 개발(AI 코드 생성 포함)의 기준을 세웁니다.

> **v1.1 변경 요약**: 화면(컴포넌트)이 `songs.json`을 직접 읽지 않고, **데이터 접근 함수(예: `getSongs()`)를 통해 비동기로 접근**하도록 구조를 확정했습니다. 지금은 이 함수가 로컬 JSON을 읽어 흉내만 내지만("API Mocking"), 나중에 실제 API/DB(Supabase 등)로 바뀌어도 **화면 코드는 그대로 두고 이 함수 내부만 교체**하면 되도록 하기 위함입니다.

## 1. 전체 구조 한눈에 보기 (Architecture)

```
[사용자 스마트폰]
      │  (웹 접속)
      ▼
[Next.js + React] ── 화면을 그림 (목록/상세)
      │
      ├─ [Tailwind CSS] ── 모바일 친화 디자인
      │
      ├─ [데이터 접근 함수] (lib/api) ── getSongs(), getSongById() 등
      │        │  (지금은 아래 JSON을 읽어 흉내, 나중엔 내부만 교체)
      │        ▼
      │   [songs.json] ── 응원가 데이터 (제목/영상/가사)
      │
      └─ [YouTube iframe] ── 영상 재생 (임베드)

배포: [Vercel] ── 위 결과물을 인터넷에 올려줌
확장 시: [Supabase] ── 데이터가 많아지면 JSON 대신 사용 (데이터 접근 함수 내부만 교체)
```

> 핵심만 말하면: **Next.js로 화면을 그리고, Tailwind로 꾸미고, (API 형태의) 데이터 접근 함수를 통해 JSON 파일에서 데이터를 읽어와, 유튜브 영상을 끼워 넣고, Vercel로 인터넷에 올린다**입니다.

## 2. 스택 구성 (Stack Details)

### 2.1 프레임워크 — Next.js (+ React)
- **역할**: 웹 화면을 만드는 뼈대. 목록 화면, 상세 화면을 만듭니다.
- **쉬운 설명**: 화면을 **레고 블록(컴포넌트)** 처럼 조각내서 만들 수 있게 해주는 도구입니다. '응원가 카드' 블록을 하나 만들면, 곡 개수만큼 그 블록을 찍어낼 수 있습니다.
- **왜 이걸 쓰나**:
  - 화면 이동(목록 ↔ 상세)이 부드럽게 처리됩니다. (F-01 → F-02 이동)
  - 초보자용 자료와 AI 지원이 풍부해 바이브코딩과 궁합이 좋습니다.

### 2.2 스타일 — Tailwind CSS
- **역할**: 색, 크기, 여백 등 **화면을 예쁘고 편하게 꾸미는** 도구.
- **쉬운 설명**: 화면 요소에 '색칠 스티커'를 붙이듯 스타일을 입히는 방식입니다. 별도 디자인 파일을 오갈 필요 없이 화면 코드에 바로 적용합니다.
- **왜 이걸 쓰나**:
  - **모바일 반응형**을 빠르게 잡기 좋습니다. (F-03 모바일 친화 레이아웃)
  - 영상 16:9 비율 유지, 터치 영역 크기 조절 같은 세부 규칙을 간단히 적용할 수 있습니다.

### 2.3 데이터 — JSON 파일 (`songs.json`)
- **역할**: 응원가 정보(제목, 영상, 가사)를 담아두는 **데이터 창고**.
- **쉬운 설명**: 엑셀 표 한 장을 파일로 만들어 둔 것과 비슷합니다. 응원가 한 곡이 한 줄(하나의 데이터)입니다.
- **왜 이걸 쓰나 (MVP 단계)**:
  - 곡 수가 적은 초기에는 **DB 없이 파일 하나로 충분**합니다. (가장 단순하고 비용 0)
  - 기능 명세서의 `videos`, `tags`, `lyrics` 구조를 그대로 담습니다.
- **주의**: 이 단계에서는 **Supabase가 필요 없습니다.** (2.6 참고)

### 2.3.1 데이터 접근 방식 — API Mocking *(v1.1 추가)*
- **역할**: 화면(컴포넌트)이 `songs.json`을 **직접 import해서 읽지 않고**, `lib/api/songs.ts`의 함수(예: `getSongs()`, `getSongById(id)`)를 통해 **비동기(async)로 접근**합니다.
- **쉬운 설명**: 지금은 이 함수 내부가 그냥 로컬 JSON 파일을 읽어서 돌려주지만, 화면 입장에서는 "어딘가에 물어보고(await) 응답을 받는" **진짜 API를 호출하는 것과 똑같은 모양**으로 씁니다. 즉, 실제 API가 없는 지금 단계에서 **API를 흉내(Mock)** 내는 것입니다.
- **왜 이걸 쓰나**:
  - **나중에 실제 API/DB(Supabase 등)로 바뀔 때**, `lib/api/songs.ts` 내부 구현만 바꾸면 되고 **화면 코드는 거의 그대로** 유지됩니다. (2.6 확장 시나리오와 바로 연결)
  - F-01의 "로딩 중" 상태(기능명세서 참고)가 자연스럽게 재현됩니다. 로컬 JSON은 사실상 즉시 읽히지만, 함수가 `Promise`를 반환하는 형태를 유지해 실제 네트워크 호출과 같은 흐름(로딩 → 성공/실패)으로 화면을 짤 수 있습니다.
  - 나중에 컴포넌트 단위 테스트를 작성할 때도, `lib/api/songs.ts`만 대체(mock)하면 되어 테스트가 쉬워집니다.
- **주의**: 이것은 **새로운 서버나 네트워크 통신을 추가하는 것이 아닙니다.** 여전히 별도 백엔드 서버 없이, 함수 호출 방식만 "API스럽게" 통일하는 것입니다. (3번 기술 스택 고정 원칙 위반 아님)

### 2.4 영상 — YouTube 임베드 (iframe)
- **역할**: 공식/현장 영상을 화면 안에서 재생.
- **쉬운 설명**: 영상을 내 서버에 복사해 오는 게 아니라, **유튜브에 있는 원본을 창문처럼 비춰 보여주는** 방식입니다.
- **왜 이걸 쓰나**:
  - 저작권 측면에서 안전합니다. (PRD 9번 유의사항)
  - `videos` 목록의 `youtubeId`만 있으면 바로 영상을 띄울 수 있습니다. (F-04)

### 2.5 배포 — Vercel
- **역할**: 완성된 웹을 **인터넷에 올려** 누구나 접속할 수 있게 합니다.
- **쉬운 설명**: 내 컴퓨터 안에만 있던 결과물을 '가게 오픈'하듯 세상에 공개하는 단계입니다.
- **왜 이걸 쓰나**:
  - Next.js를 만든 회사의 서비스라 연결이 매끄럽습니다.
  - 개인/소규모는 **무료 요금제로 시작** 가능합니다.

### 2.6 (확장용) 데이터베이스 — Supabase
- **역할**: 곡이 아주 많아지거나 관리자 기능(F-14)이 필요해질 때, JSON 파일을 대신하는 **진짜 데이터베이스**.
- **중요**: **MVP에서는 사용하지 않습니다.** 지금 도입하면 오히려 복잡해집니다.
- **언제 도입하나**: 확장 기능(검색 F-11, 관리자 F-14 등)을 본격적으로 만들 때 넘어가면 됩니다. 데이터 구조를 미리 맞춰뒀기 때문에 그때 옮겨도 무리가 없습니다.

## 3. 기능 ↔ 기술 연결표 (Feature-Tech Mapping)

> 기능 명세서의 각 기능이 어떤 기술로 구현되는지 연결한 표입니다.

| 기능 | 주로 사용하는 기술 |
|------|------------------|

## 4. 폴더/파일 구조 예시 (Project Structure)

> 실제 코드를 만들 때 이런 모양이 됩니다. (참고용, 도구에 따라 달라질 수 있음)

```
incheon-chants/
├── data/
│   └── songs.json        ← 응원가 데이터 (핵심)
├── lib/
│   └── api/
│       └── songs.ts      ← 데이터 접근 함수 (getSongs, getSongById) — API Mocking 계층
├── app/ (또는 pages/)
│   ├── page.jsx          ← 목록 화면 (F-01)
│   └── songs/[id]/
│       └── page.jsx      ← 상세 화면 (F-02, F-04)
├── components/
│   ├── SongCard.jsx      ← 응원가 카드 블록
│   ├── VideoTabs.jsx     ← 공식/현장 영상 선택 탭 (F-04)
│   └── YoutubePlayer.jsx ← 유튜브 영상 재생
├── package.json          ← 사용하는 도구 목록
└── ...
```

- **`data/songs.json`이 가장 중요합니다.** 여기만 채워 넣으면 화면이 자동으로 그려지도록 만드는 것이 목표입니다.
- **화면(`app/`, `components/`)은 `data/songs.json`을 직접 읽지 않고, 항상 `lib/api/songs.ts`를 거칩니다.** (2.3.1 참고)

## 6. 비용 구조 (Cost) — 단계별

| 단계 | 사용 도구 | 예상 비용 |
|------|----------|-----------|
| MVP | Next.js, Tailwind, JSON, Vercel(무료 요금제) | **0원** |
| 확장 초기 | 위 + Supabase(무료 요금제) | 대체로 0원 (사용량 적을 때) |
| 확장 이후 | 트래픽·데이터 증가 시 유료 전환 가능 | 사용량에 따라 |

> MVP는 **비용 0원**으로 만들고 공개할 수 있습니다. 유료 전환은 서비스가 커진 뒤 고민해도 늦지 않습니다.

---

## 7. 이 단계에서 쓰지 않는 것 (Out of Scope)

> 초보 단계에서 흔히 '있어 보여서' 넣었다가 복잡해지는 것들입니다. **MVP에서는 의도적으로 제외**합니다.

- 로그인/회원 관리 도구 (PRD Non-Goals)
- 별도 백엔드 서버 (JSON + Next.js로 충분)
- 상태 관리 라이브러리(Redux 등) — 화면이 단순해 불필요
- 데이터베이스(Supabase) — 확장 시점까지 보류

---

## 8. 주요 의사결정 기록 (Decision Log)

| 결정 | 선택 | 이유 |
|------|------|------|
| 초기 데이터 저장 방식 | **JSON 파일** (DB 아님) | 곡 수가 적어 파일로 충분, 비용·복잡도 최소화 |
| 영상 재생 방식 | **YouTube 임베드** | 저작권 안전, 구현 간단 |
| DB(Supabase) 도입 시점 | **확장 단계로 보류** | MVP에 불필요한 복잡도 방지 |
| 배포 도구 | **Vercel** | Next.js와 궁합·무료 시작 |
| 데이터 접근 방식 *(v1.1)* | 컴포넌트의 **직접 JSON import(X)** → **`lib/api/songs.ts` 함수 경유(O)**, 즉 API Mocking | 실제 API/DB로 전환 시 화면 코드 수정 없이 함수 내부만 교체하기 위함. 별도 서버·네트워크 통신을 추가하는 것은 아님 |
