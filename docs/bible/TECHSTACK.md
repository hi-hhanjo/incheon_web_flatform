## 1. 전체 구조 한눈에 보기 (Architecture)

```
[사용자 스마트폰]
      │  (웹 접속)
      ▼
[Next.js + React] ── 화면을 그림 (목록/상세/구단 정보)
      │
      ├─ [Tailwind CSS] ── 모바일 친화 디자인
      │
      ├─ [데이터 접근 함수] (lib/api) ── getSongs(), getStandings() 등
      │        │  (JSON을 읽어 API처럼 돌려줌. 나중엔 내부만 교체)
      │        ▼
      │   [data/*.json] ── 응원가 + 구단 데이터
      │        ▲
      │        │ (파일만 주고받음 — 앱 코드는 건드리지 않음)
      │   [크롤러 사이드카] (scraper/, Python) ── 2.7 참고
      │        │
      │        └─ 다음 스포츠 / 구단 공식 홈페이지 / 인천네이션
      │
      └─ [YouTube iframe] ── 영상 재생 (임베드)

배포: [Vercel] ── 위 결과물을 인터넷에 올려줌 (앱은 외부 API 키가 필요 없음)
자동화: [GitHub Actions] ── 주 2회 크롤러를 돌려 data/*.json을 갱신·커밋
확장 시: [Supabase] ── 데이터가 많아지면 JSON 대신 사용 (데이터 접근 함수 내부만 교체)
```

> **크롤러는 앱의 일부가 아닙니다.** 화살표가 `data/*.json`에서 끊기는 게 핵심입니다 — 크롤러는
> 파일을 만들어 놓고 빠지고, 앱은 그 파일을 읽을 뿐 서로를 호출하지 않습니다. 그래서 크롤러가
> Python이어도 웹앱 스택(Next.js)은 그대로입니다.

> 핵심만 말하면: **Next.js로 화면을 그리고, Tailwind로 꾸미고, (API 형태의) 데이터 접근 함수를 통해 JSON 파일에서 데이터를 읽어와, 유튜브 영상을 끼워 넣고, Vercel로 인터넷에 올린다**입니다.

## 2. 스택 구성 (Stack Details)

### 2.1 프레임워크 — Next.js (+ React)
- **역할**: 웹 화면을 만드는 뼈대. 목록 화면, 상세 화면을 만듭니다.
- **쉬운 설명**: 화면을 **레고 블록(컴포넌트)** 처럼 조각내서 만들 수 있게 해주는 도구입니다. '응원가 카드' 블록을 하나 만들면, 곡 개수만큼 그 블록을 찍어낼 수 있습니다.
- **왜 이걸 쓰나**:
  - 화면 이동(목록 ↔ 상세)이 부드럽게 처리됩니다. (F-01 → F-02 이동)
  - 초보자용 자료와 AI 지원이 풍부해 바이브코딩과 궁합이 좋습니다.

### 2.2 스타일 — Tailwind CSS
- **역할**: 색, 크기, 여백 등 **화면을 예쁘고 편하게 꾸미는** 도구.
- **쉬운 설명**: 화면 요소에 '색칠 스티커'를 붙이듯 스타일을 입히는 방식입니다. 별도 디자인 파일을 오갈 필요 없이 화면 코드에 바로 적용합니다.
- **왜 이걸 쓰나**:
  - **모바일 반응형**을 빠르게 잡기 좋습니다. (F-03 모바일 친화 레이아웃)
  - 영상 16:9 비율 유지, 터치 영역 크기 조절 같은 세부 규칙을 간단히 적용할 수 있습니다.

### 2.3 데이터 — JSON 파일 (`songs.json`)
- **역할**: 응원가 정보(제목, 영상, 가사)를 담아두는 **데이터 창고**.
- **쉬운 설명**: 엑셀 표 한 장을 파일로 만들어 둔 것과 비슷합니다. 응원가 한 곡이 한 줄(하나의 데이터)입니다.
- **왜 이걸 쓰나 (MVP 단계)**:
  - 곡 수가 적은 초기에는 **DB 없이 파일 하나로 충분**합니다. (가장 단순하고 비용 0)
  - 기능 명세서의 `videos`, `tags`, `lyrics` 구조를 그대로 담습니다.
- **주의**: 이 단계에서는 **Supabase가 필요 없습니다.** (2.6 참고)

### 2.3.1 데이터 접근 방식 — API Mocking *(v1.1 추가)*
- **역할**: 화면(컴포넌트)이 `songs.json`을 **직접 import해서 읽지 않고**, `lib/api/songs.ts`의 함수(예: `getSongs()`, `getSongById(id)`)를 통해 **비동기(async)로 접근**합니다.
- **쉬운 설명**: 지금은 이 함수 내부가 그냥 로컬 JSON 파일을 읽어서 돌려주지만, 화면 입장에서는 "어딘가에 물어보고(await) 응답을 받는" **진짜 API를 호출하는 것과 똑같은 모양**으로 씁니다. 즉, 실제 API가 없는 지금 단계에서 **API를 흉내(Mock)** 내는 것입니다.
- **왜 이걸 쓰나**:
  - **나중에 실제 API/DB(Supabase 등)로 바뀔 때**, `lib/api/songs.ts` 내부 구현만 바꾸면 되고 **화면 코드는 거의 그대로** 유지됩니다. (2.6 확장 시나리오와 바로 연결)
  - F-01의 "로딩 중" 상태(기능명세서 참고)가 자연스럽게 재현됩니다. 로컬 JSON은 사실상 즉시 읽히지만, 함수가 `Promise`를 반환하는 형태를 유지해 실제 네트워크 호출과 같은 흐름(로딩 → 성공/실패)으로 화면을 짤 수 있습니다.
  - 나중에 컴포넌트 단위 테스트를 작성할 때도, `lib/api/songs.ts`만 대체(mock)하면 되어 테스트가 쉬워집니다.
- **주의**: 이것은 **새로운 서버나 네트워크 통신을 추가하는 것이 아닙니다.** 여전히 별도 백엔드 서버 없이, 함수 호출 방식만 "API스럽게" 통일하는 것입니다. (3번 기술 스택 고정 원칙 위반 아님)

### 2.3.2 계약과 데이터 읽기를 분리한다 — `songs.schema.ts` *(v1.3 추가)*

- **규칙**: `lib/api/*.ts`(데이터를 읽는 코드)는 `data/*.json`을 import합니다. 반면 **타입과 데이터 값 상수는 `songs.schema.ts`에 두고, 이 파일은 JSON을 import하지 않습니다.** 클라이언트 컴포넌트(`"use client"`)가 값을 가져와야 하면 **반드시 schema에서** 가져옵니다.
- **왜 이걸 쓰나 (실제로 겪은 문제)**: `UNUSED_TAG` 같은 상수가 `songs.ts`(맨 윗줄에서 `songs.json`을 import)에 있었고, 클라이언트 컴포넌트인 `SongCard`·`SongList`가 그 값을 가져갔습니다. 그러자 **값 하나 때문에 모듈 전체가 클라이언트 번들로 끌려와 가사 전체(약 20KB)가 JS 번들에 중복 포함**됐습니다 — 이미 서버가 HTML로 보낸 데이터를 한 번 더 내려보낸 셈입니다. 계약을 분리해 없앴습니다.
- **타입만 쓰는 건 원래 안전합니다.** `import type`은 컴파일 시 지워져 번들에 남지 않습니다. 문제가 되는 건 **값**(상수·함수) import뿐입니다.
- **왜 상수를 화면에 하드코딩하지 않나**: `"미사용"` 같은 문자열은 크롤러(`classify_chants.py`)가 쓰고 화면이 읽는 **데이터 계약**입니다. 양쪽에 흩어 놓으면 한쪽만 바뀌어도 조용히 어긋납니다.
- **점검 방법**: 빌드 후 `.next/static/`에서 가사 한 소절을 검색해 나오면 안 됩니다.

### 2.4 영상 — YouTube 임베드 (iframe)
- **역할**: 공식/현장 영상을 화면 안에서 재생.
- **쉬운 설명**: 영상을 내 서버에 복사해 오는 게 아니라, **유튜브에 있는 원본을 창문처럼 비춰 보여주는** 방식입니다.
- **왜 이걸 쓰나**:
  - 저작권 측면에서 안전합니다. (PRD 9번 유의사항)
  - `videos` 목록의 `youtubeId`만 있으면 바로 영상을 띄울 수 있습니다. (F-04)

### 2.5 배포 — Vercel
- **역할**: 완성된 웹을 **인터넷에 올려** 누구나 접속할 수 있게 합니다.
- **쉬운 설명**: 내 컴퓨터 안에만 있던 결과물을 '가게 오픈'하듯 세상에 공개하는 단계입니다.
- **왜 이걸 쓰나**:
  - Next.js를 만든 회사의 서비스라 연결이 매끄럽습니다.
  - 개인/소규모는 **무료 요금제로 시작** 가능합니다.

### 2.6 (확장용) 데이터베이스 — Supabase
- **역할**: 곡이 아주 많아지거나 화면에서 직접 쓰기(등록/수정)가 필요해질 때, JSON 파일을 대신하는 **진짜 데이터베이스**.
- **중요**: **현재는 사용하지 않습니다.** 지금 도입하면 오히려 복잡해집니다. (앱은 `data/*.json`을 직접 읽습니다)
- **언제 도입하나**: 쓰기 기능이 본격적으로 필요해질 때 넘어가면 됩니다. 데이터 구조를 미리 맞춰뒀기 때문에 그때 옮겨도 무리가 없습니다.

### 2.7 데이터 수집 — 크롤러 사이드카 (Python) *(v1.2 신설 — 소급 문서화)*

> **먼저 밝힐 것**: 이 항목은 **코드가 문서를 앞서간 경우**입니다. 크롤러는 2026-07-16~17에 걸쳐
> 만들어졌는데 이 문서에 반영되지 않아, "TECHSTACK에 정의된 기술만 사용한다"(rules.md 3장)는
> 규칙과 실제 코드가 어긋나 있었습니다. 2026-07-17 현실에 맞춰 소급 기록합니다.

- **역할**: 앱이 읽을 `data/*.json`을 만들어 두는 **별도 프로그램**. 응원가 가사·영상(구단 공식
  홈페이지·인천네이션)과 구단 데이터(순위표·경기·상대 폼·상대전적 — 다음 스포츠)를 긁어옵니다.
- **쉬운 설명**: 웹사이트를 사람 대신 열어 보고 필요한 내용만 **파일로 정리해 두는 심부름꾼**입니다.
  심부름꾼은 파일을 놓고 갈 뿐, 웹페이지가 손님을 맞는 동안엔 아무 일도 하지 않습니다.
- **사용 기술**: Python 3.12 + `requests`(HTTP) + `beautifulsoup4`(HTML 파싱). 의존성은
  `scraper/requirements.txt` 두 줄이 전부입니다. *(구단 데이터는 JSON API라 `requests`만으로 되고,
  `beautifulsoup4`는 HTML 페이지인 응원가·선수단 크롤에만 씁니다.)*
- **왜 Python인가**:
  - 크롤링·HTML 파싱은 Python 생태계가 가장 단순합니다(`requests` + `beautifulsoup4` 두 줄 의존성).
  - **앱과 언어를 맞출 이유가 없습니다.** 둘은 서로를 import하지 않고 **파일로만** 만납니다(1장 다이어그램).
    앱 번들에 크롤러 코드가 섞이지 않아 배포 결과물이 가벼워집니다.
  - GitHub Actions에서 `setup-python` 한 줄로 돌아갑니다(무료).
- **왜 이게 "별도 백엔드 서버 추가"가 아닌가** (7장 Out of Scope 위반이 아닌 이유):
  - **상시 떠 있는 서버가 아닙니다.** 주 2회 GitHub Actions에서 몇 초 돌고 끝나며, 사용자의 요청을
    받지 않습니다. 사용자는 크롤러의 존재를 알 수 없습니다.
  - 앱이 크롤러를 **호출하지 않습니다.** 크롤러가 커밋한 JSON이 다음 배포에 실려 갈 뿐입니다.
  - 그래서 Vercel에는 여전히 **Next.js 하나만** 배포되고, 운영 비용도 0원 그대로입니다(6장).
- **원칙**: 크롤러는 `data/*.json`을 **앱 스키마 그대로** 뱉는 것까지만 책임집니다. 화면·컴포넌트·
  `lib/api`를 건드리지 않습니다. 상세 규격은 **CRAWLER_SPEC.md**에 있습니다.
- **응원가는 사람 검수를 거칩니다**: 순위표 같은 기계적 사실과 달리 가사·영상 선별은 판단이 필요해,
  크롤러가 검수 큐(`data/songs-candidates.json`)까지만 만들고 사람이 승격시킵니다
  (CRAWLER_SPEC.md 3.7).

### 2.8 자동화 — GitHub Actions *(v1.2 신설)*

- **역할**: 주 2회(월·목 06:00 KST) 구단 데이터 크롤러를 돌려 바뀐 `data/*.json`을 커밋합니다.
- **왜 이걸 쓰나**: 이미 GitHub에 코드가 있고(5장), 크론 실행이 무료입니다. 별도 서버가 필요 없습니다.
- 응원가 크롤러는 여기 없습니다 — 1년에 몇 곡 수준이라 필요할 때 수동 실행합니다.

## 3. 기능 ↔ 기술 연결표 (Feature-Tech Mapping)

> 기능 명세서의 각 기능이 어떤 기술로 구현되는지 연결한 표입니다.

| 기능 | 주로 사용하는 기술 |
|------|------------------|
| F-01 목록 화면 | Next.js(화면) + Tailwind(스타일) + songs.json(데이터) |
| F-02 상세 화면 | Next.js(화면 이동/데이터 찾기) + Tailwind |
| F-04 영상 선택 탭 | Next.js(탭 상태 관리) + YouTube iframe |
| F-03 모바일 레이아웃 | Tailwind (반응형) |
| F-11 검색 | Next.js(클라이언트 필터) + `lib/search.ts`(순수 함수) — 별도 기술 없음 *(v1.4)* |

## 4. 폴더/파일 구조 예시 (Project Structure)

> *(v1.2 — 실제 구조로 갱신했습니다. 이전 버전은 응원가만 있던 초기 예시라 `scraper/`·`docs/`가
> 없었고, rules.md 3장이 "폴더 구조는 이 절을 따른다"고 가리키는 대상이 현실과 달랐습니다.)*

```
incheon_chant_web/
├── app/                        ← 화면 (Next.js App Router)
│   ├── page.tsx                  목록 (F-01)
│   ├── songs/[id]/page.tsx       상세 (F-02, F-04)
│   └── club/                     구단 정보 (F-16~F-18)
│       ├── page.tsx              경기·다가오는 매치
│       ├── standings/page.tsx    순위표
│       └── opponent/page.tsx     다음 상대
├── components/                 ← 화면 부품 (docs/design/components/*.md 명세)
│   ├── SongCard.tsx, VideoTabs.tsx, Lyrics.tsx, SourceNote.tsx …
│   └── club/                     구단 정보 전용 부품
├── lib/
│   ├── api/                    ← 데이터 접근 계층 (2.3.1) — 화면은 반드시 여기를 거친다
│   │   ├── songs.ts, standings.ts, matches.ts, headToHead.ts, opponentScouting.ts
│   │   │                         data/*.json을 import해 읽는다 (서버 전용)
│   │   ├── songs.schema.ts       응원가 타입·상수 — **data/*.json을 import하지 않는다**(2.3.2)
│   │   ├── teams.ts              K리그1 구단 메타데이터(엠블럼·팀 페이지 URL) — data/*.json 미참조
│   │   └── sources.ts            데이터 수집처 상수 (출처 표기용)
│   ├── format.ts                 날짜 등 표시용 순수 함수
│   └── search.ts                 F-11 검색의 순수 함수 (걸러내기·가사 줄 찾기·강조 조각)
├── data/                       ← 런타임 데이터 창고 (앱이 직접 import)
│   ├── songs.json                응원가 (핵심)
│   ├── standings.json, matches.json, opponent-forms.json, head-to-head.json
│   ├── *-meta.json               각 데이터의 수집 기준일
│   └── standings-history/        순위표 수집일별 이력
├── scraper/                    ← 크롤러 사이드카 (2.7, Python) — 앱과 독립
│   ├── kleague/                  구단 데이터 파서
│   ├── chants/                   응원가·선수단 파서
│   ├── export_*.py               크롤 → data/*.json
│   └── requirements.txt
├── docs/                       ← 근거 문서 (rules.md 1장의 '정답의 기준')
│   ├── bible/                    PRD · FUNCTION · TECHSTACK
│   ├── design/                   BASE + components/*.md
│   └── data/                     API_SPEC · ERD_SPEC · CLUB_ERD_SPEC · CRAWLER_SPEC
├── .agent/rules.md             ← 프로젝트 규칙 (CLAUDE.md가 import)
├── .github/workflows/          ← 주간 데이터 갱신 (2.8)
└── package.json
```

- **`data/`가 가장 중요합니다.** 여기만 채워 넣으면 화면이 자동으로 그려지도록 만드는 것이 목표입니다.
- **화면(`app/`, `components/`)은 `data/*.json`을 직접 읽지 않고, 항상 `lib/api/`를 거칩니다.** (2.3.1)
- **`lib/` 바로 아래(`format.ts`·`search.ts`)는 데이터 접근이 아니라 '순수 함수' 자리입니다.** `data/*.json`을 import하지 않고 넘겨받은 값만 다룹니다 — 그래서 클라이언트 컴포넌트가 가져다 써도 데이터가 번들로 딸려 오지 않습니다(`songs.schema.ts`와 같은 이유, 2.3.2). 화면 부품이 문자열을 직접 자르거나 비교하지 않고 여기에 맡깁니다.
- **`scraper/`는 `data/`에만 씁니다.** 앱 코드(`app/`·`components/`·`lib/`)를 건드리지 않습니다.
- **`scripts/` 디렉터리는 두지 않습니다** — 크롤링·데이터 도구는 전부 `scraper/`에 모읍니다.
  *(v1.2 — 과거 `scripts/`에 일회성 API 검증 스크립트가 있어 도구가 두 군데로 갈렸고, 2026-07-17
  해당 스크립트를 제거하며 디렉터리도 없앴습니다.)*

## 5. 개발 준비물 (Prerequisites)

개발을 시작하기 전에 컴퓨터에 준비되어야 할 것들입니다.

- **Node.js**: Next.js를 돌리기 위한 기본 엔진. (최신 안정 버전 설치 권장)
- **코드 편집기**: VS Code 등
- **GitHub 계정**: 코드를 보관하고 Vercel과 연결할 때 사용. 주간 데이터 갱신(2.8)도 여기서 돌아갑니다
- **Vercel 계정**: 배포용 (GitHub 계정으로 가입 가능)
- **Python 3.12+** *(v1.2 추가)*: **크롤러(2.7)를 직접 돌릴 때만** 필요합니다. 웹앱을 개발·배포하는 데는
  필요 없습니다 — 크롤러가 만든 `data/*.json`이 이미 커밋돼 있고, 정기 갱신은 GitHub Actions가
  대신합니다. 직접 돌리려면: `pip install -r scraper/requirements.txt`
- **외부 API 키: 필요 없습니다** *(v1.2)*. 예전엔 상대전적 때문에 `API_FOOTBALL_KEY`가 필요했지만
  2026-07-17 크롤로 전환하며 없앴습니다 — `.env` 없이 클론 후 바로 `npm run dev`가 됩니다.

> 위 도구들의 정확한 설치 방법과 버전은 수강 중인 과정의 안내를 따르시는 것을 권장합니다.

## 6. 비용 구조 (Cost) — 단계별

| 단계 | 사용 도구 | 예상 비용 |
|------|----------|-----------|
| MVP | Next.js, Tailwind, JSON, Vercel(무료 요금제) | **0원** |
| 현재 *(v1.2)* | 위 + 크롤러(Python·자체 실행) + GitHub Actions(공개 레포 무료) | **0원** |

> MVP는 **비용 0원**으로 만들고 공개할 수 있습니다. 유료 전환은 서비스가 커진 뒤 고민해도 늦지 않습니다.
> 크롤러를 붙이고도 **0원이 유지됩니다** — 크롤 대상이 모두 공개 페이지·공개 API라 유료 API 구독이
> 없고, 실행은 주 2회 GitHub Actions 몇 초뿐입니다. *(2026-07-17 유일한 유료 후보였던
> API-Football 무료 티어 의존마저 제거했습니다 — 2.7.)*

## 7. 이 단계에서 쓰지 않는 것 (Out of Scope)

> 초보 단계에서 흔히 '있어 보여서' 넣었다가 복잡해지는 것들입니다. **MVP에서는 의도적으로 제외**합니다.

- 로그인/회원 관리 도구 (PRD Non-Goals)
- **별도 백엔드 서버** (JSON + Next.js로 충분) — *(v1.2 명확화: 크롤러(2.7)는 여기 해당하지 않습니다.
  상시 떠서 사용자 요청을 받는 서버가 아니라, 주 2회 몇 초 돌고 파일만 남기고 끝나는 오프라인
  도구입니다. Vercel에는 여전히 Next.js 하나만 배포됩니다.)*
- 상태 관리 라이브러리(Redux 등) — 화면이 단순해 불필요
- 데이터베이스(Supabase) — 확장 시점까지 보류
- **유료 외부 데이터 API** *(v1.2 추가)* — 공개 페이지 크롤로 충분합니다. 앱 런타임이 외부 API를
  호출하지 않으므로 API 키·`.env`도 필요 없습니다.

## 8. 주요 의사결정 기록 (Decision Log)

| 결정 | 선택 | 이유 |
|------|------|------|
| 초기 데이터 저장 방식 | **JSON 파일** (DB 아님) | 곡 수가 적어 파일로 충분, 비용·복잡도 최소화 |
| 영상 재생 방식 | **YouTube 임베드** | 저작권 안전, 구현 간단 |
| DB(Supabase) 도입 시점 | **확장 단계로 보류** | MVP에 불필요한 복잡도 방지 |
| 배포 도구 | **Vercel** | Next.js와 궁합·무료 시작 |
| 데이터 접근 방식 *(v1.1)* | 컴포넌트의 **직접 JSON import(X)** → **`lib/api/songs.ts` 함수 경유(O)**, 즉 API Mocking | 실제 API/DB로 전환 시 화면 코드 수정 없이 함수 내부만 교체하기 위함. 별도 서버·네트워크 통신을 추가하는 것은 아님 |
| 타입·상수의 위치 *(v1.3)* | 데이터 읽는 모듈에 함께(X) → **`songs.schema.ts`로 분리(O)** | 클라이언트 컴포넌트가 상수 하나를 가져가면 JSON을 import하는 모듈 전체가 딸려와 **가사 20KB가 번들에 중복** 포함됐음. 계약(타입·상수)과 데이터 읽기를 분리해 해결 (2.3.2) |
| 데이터 수집 방식 *(v1.2)* | 유료/무료 스포츠 API(X) → **공개 페이지 크롤 사이드카(O)** | 무료 API 티어가 K리그 현재 시즌을 막아 실연동이 불가능했음. 공개 API·페이지 크롤은 제한이 없고 비용 0. CRAWLER_SPEC.md 0장 |
| 크롤러 언어 *(v1.2)* | 앱과 통일(TypeScript)(X) → **Python(O)** | 앱과 파일로만 만나 언어를 맞출 이유가 없음. 크롤링·HTML 파싱 생태계가 단순하고(의존성 2개), 앱 번들에 섞이지 않음 |
| 크롤러↔앱 결합 *(v1.2)* | 앱이 크롤러 호출(X) → **`data/*.json` 파일로만 전달(O)** | 앱 런타임에서 크롤이 일어나지 않아 Vercel 배포·비용·안정성이 그대로. 크롤러가 죽어도 화면은 마지막 스냅샷으로 동작 |
| 상대전적 데이터 *(v1.2)* | API-Football 실시간 조회(X) → **다음 크롤 스냅샷(O)** | 유일한 외부 API 의존이라 키가 없으면 화면이 조용히 비었고 팀명 매핑이 중복됐음. 전환 전 두 소스의 날짜·스코어 일치를 검증. CRAWLER_SPEC.md 3.6 |
