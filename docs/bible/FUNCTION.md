## F-01. 응원가 목록 화면

### 개요
- 인천 유나이티드 응원가들을 카드 목록으로 보여주고, 하나를 선택하면 상세 화면으로 넘겨줍니다.

### 화면 구성 요소
- **상단 제목**: 예) "인천 유나이티드 응원가"
- **검색창(SearchBar)**: 제목·가사로 목록을 좁힌다 (곡이 1개 이상일 때만 표시). 상세는 F-11.
- **응원가 카드 목록**: 세로로 나열되는 카드들
  - 카드에 표시: 응원가 **제목** (필수)
  - (선택) '대표곡' 표시(배지) — 있으면 눈에 띄게
  - (선택) **'미사용' 표시(배지)** — 팀을 떠난 선수의 응원가 *(v1.3 신설)*

### 동작 흐름
1. 사용자가 웹에 접속한다.
2. `songs.json`의 응원가 데이터를 읽어 목록을 그린다.
3. **정렬** *(v1.5 — 단일 정렬로 확정)*: **분류 순서**로 표시한다 — **팀 응원가 → 선수 응원가(현역) → 미사용**, 각 묶음 안에서는 제목 가나다순(한글 먼저, 영어 나중). **사용자가 정렬을 바꾸는 수단은 없다.**
4. 사용자가 카드를 탭하면 → 해당 곡의 **상세 화면(F-02)** 으로 이동한다. (곡의 `id`를 넘긴다)

> **왜 '등록 순서'를 버렸나** *(v1.3)*: 응원가를 크롤링으로 수집하면서 '등록 순서'가 사실상 **크롤러가 정렬한 순서**가 되어 사람이 의도한 순서가 아니게 됐습니다(게다가 코드포인트 정렬이라 영어 제목이 한글보다 앞섰습니다). 또 팀을 떠난 선수의 응원가가 목록 위쪽에 섞여, 핵심 타겟인 신규팬이 **지금 불리지도 않는 곡을 먼저 배우는** 문제가 있었습니다.
>
> **주관성 문제는 재발하지 않습니다**: v1.1이 '대표곡 우선 정렬'을 뺀 이유는 대표곡 판단이 주관적이어서였습니다. 새 기준인 `category`(팀/선수)와 `미사용`은 **구단 공식 선수단 명단**으로 기계가 판정합니다(`scraper/classify_chants.py`) — 사람의 취향이 개입하지 않습니다.
>
> **왜 '가나다순' 토글을 없앴나** *(v1.5)*: 실제 데이터에서 **두 모드가 거의 같은 목록을 냈기 때문**입니다. 34곡의 분류는 팀 응원가 28곡 / 선수 응원가 현역 **1곡**(무고사) / 미사용 5곡인데, 두 모드 모두 묶음 안이 가나다순이고 미사용이 맨 아래로 고정이라 **차이가 나는 것은 현역 선수곡 1곡의 위치뿐**이었습니다. 나머지 33곡은 자리가 같아, 사용자는 자기가 어느 모드에 있는지 알 수 없었습니다 — 44px 터치 영역 두 개를 쓰면서 선택지를 주지 못하는 상태였습니다. **정렬 규칙(위 3번)은 그대로 두고 토글만 걷어냅니다.**
>
> **되살릴 조건**: `category`의 팀/선수 구분이 정렬 기준으로 의미를 가지려면 **현역 선수 응원가가 여러 곡**이어야 합니다. 선수단 변동으로 현역 선수곡이 늘어 두 묶음의 크기가 비슷해지면, 그때 이 결정을 다시 보세요(`scraper/classify_chants.py`가 분류를 갱신합니다).

### 상태별 처리
- **정상**: 곡이 1개 이상 있으면 검색창과 카드 목록을 보여준다.
- **빈 상태**: 곡이 하나도 없으면 "등록된 응원가가 없습니다" 문구만 보여준다. (검색창은 표시하지 않는다 — 검색할 대상이 없다)
- **로딩 중**: 데이터를 읽는 동안 간단한 로딩 표시(예: "불러오는 중...")를 보여준다.

### 완료 조건 (체크리스트)
- [ ] 응원가 제목이 카드에 정확히 표시된다.
- [ ] 진입 시 팀 응원가 → 선수 응원가(현역) → 미사용 순으로 표시된다. *(v1.3)*
- [ ] 각 묶음 안에서 제목이 가나다순(한글 먼저, 영어 나중)으로 놓인다.
- [ ] 미사용 곡에 '미사용' 배지가 붙고, 목록 맨 아래에 있다. *(v1.3)*
- [ ] 카드를 누르면 올바른 곡의 상세 화면으로 이동한다.
- [ ] 곡이 없을 때 빈 상태 문구가 나오고, 이때는 검색창이 보이지 않는다.

## F-02. 응원가 상세 화면

### 개요
- 선택한 응원가 하나에 대해 **영상(공식/현장 선택 가능) + 가사**를 보여줍니다.

### 화면 구성 요소 (위에서부터 순서대로)
1. **뒤로 가기 버튼**: 목록 화면으로 복귀
2. **응원가 제목**
3. **영상 선택 탭**: [공식 영상] [현장 영상] (F-04에서 상세 정의)
4. **영상 영역**: 선택된 유튜브 영상 (임베드)
5. **가사 영역**: 영상 아래에 가사 표시 (줄바꿈 그대로 반영)

### 동작 흐름
1. 목록에서 넘어온 곡 `id`로 해당 곡 데이터를 찾는다.
2. 화면을 **영상 → 가사** 순서로 그린다.
3. 영상은 기본값(공식 영상)을 먼저 보여준다. (F-04)
4. 사용자가 '뒤로 가기'를 누르면 목록 화면으로 돌아간다.

### 상태별 처리
- **정상**: 영상과 가사가 모두 있으면 순서대로 표시한다.
- **가사 없음**: 가사가 비어 있으면 "가사 준비 중입니다" 문구를 보여준다.
- **잘못된 접근**: 존재하지 않는 곡 `id`로 접근하면 "해당 응원가를 찾을 수 없습니다" 문구와 목록으로 가는 버튼을 보여준다.

### 완료 조건 (체크리스트)
- [ ] 화면이 영상 → 가사 순서로 표시된다.
- [ ] 가사의 줄바꿈이 원본대로 보인다.
- [ ] 뒤로 가기로 목록에 정상 복귀한다.
- [ ] 없는 곡으로 접근 시 안내 문구가 나온다.

## F-03. 모바일 친화 레이아웃

### 개요
- 스마트폰 세로 화면을 기준으로, 모든 화면이 편하게 보이고 눌러지게 합니다.

### 세부 규칙
- **영상 크기**: 화면 너비에 맞춰 자동으로 조절되며, 가로세로 비율(16:9)을 유지한다.
- **터치 영역**: 카드·탭·버튼은 손가락으로 누르기 편한 크기(권장 최소 44px 높이)로 만든다.
- **글자 크기**: 가사는 야외에서도 읽히도록 충분히 크게(본문 권장 16px 이상).
- **가로 스크롤 금지**: 어떤 화면에서도 좌우로 밀리는 현상이 없어야 한다.

### 상태별 처리
- **화면 크기 변화**: 태블릿·PC에서 봐도 깨지지 않게 콘텐츠 폭을 적절히 제한한다.

### 완료 조건 (체크리스트)
- [ ] 스마트폰에서 영상이 화면 폭에 꽉 맞게 보인다.
- [ ] 버튼·탭이 누르기 편한 크기다.
- [ ] 가로 스크롤이 발생하지 않는다.
- [ ] 태블릿·PC에서도 레이아웃이 깨지지 않는다.

## F-04. 영상 종류 선택 (공식 / 현장)

### 개요
- 한 곡에 여러 영상(공식/현장 등)이 있을 때, 탭으로 하나를 골라 볼 수 있게 합니다.

### 화면 구성 요소
- **영상 선택 탭**: 곡의 `videos` 목록을 읽어 탭 버튼을 자동 생성
  - 예: `videos`에 official, live 2개가 있으면 → [공식 영상] [현장 영상] 2개 탭
- **선택 표시**: 현재 선택된 탭이 시각적으로 구분됨(강조)

### 동작 흐름
1. 상세 화면 진입 시, `videos` 목록의 **첫 번째 영상(기본: 공식 영상)** 을 선택 상태로 시작한다.
2. 사용자가 다른 탭(예: 현장 영상)을 누르면 → 해당 영상으로 **교체**한다.
3. 이때 **선택된 영상 1개만 불러온다.** (나머지는 불러오지 않아 로딩 부담을 줄임)

### 상태별 처리
- **영상 2개 이상**: 탭을 여러 개 보여준다.
- **영상 1개**: 탭을 만들지 않거나 1개만 보여주고, 그 영상을 바로 표시한다. (탭이 무의미하게 1개만 뜨는 걸 방지)
- **영상 0개**: "영상 준비 중입니다" 문구를 보여준다. (가사는 그대로 표시)

### 완료 조건 (체크리스트)
- [ ] `videos` 개수에 맞춰 탭이 자동 생성된다.
- [ ] 진입 시 기본값(공식 영상)이 먼저 보인다.
- [ ] 탭을 누르면 영상이 정확히 교체된다.
- [ ] 선택되지 않은 영상은 불러오지 않는다.
- [ ] 영상이 1개거나 0개인 경우가 자연스럽게 처리된다.

## F-11. 검색 기능 *(v1.4 신설)*

### 개요
- 목록 화면(F-01) 위쪽의 검색창에 글자를 입력하면, **제목 또는 가사**에 그 글자가 든 곡만 목록에 남습니다.

> **왜 지금 만드나**: PRD 10장이 검색을 '확장 1순위'로 정한 이유는 "곡이 많아지면 열성팬이 곡을 못 찾는다"였습니다. 곡이 2개일 때 이 문제는 가상이었지만, 크롤링으로 **34곡**이 되면서 실제 문제가 됐습니다. 화면을 스크롤해서 눈으로 찾는 비용이 검색창을 만드는 비용을 넘어섰습니다.

> **왜 '가사'까지 뒤지나**: PRD F-11이 "제목/가사로 검색"이라고 정해 뒀습니다. 실제로 팬은 **제목을 모른 채 가사 한 소절만 기억하는 경우**가 흔합니다("'승리의 함성' 나오는 그 노래"). 제목만 뒤지면 이 경우를 놓칩니다.

### 화면 구성 요소
- **검색창(SearchBar)**: 화면 제목과 카드 목록 사이. 안내 문구(placeholder)는 `제목·가사로 검색`
- **지우기 버튼**: 입력한 글자가 있을 때만 검색창 오른쪽에 나타남
- **가사 스니펫(LyricsSnippet)** *(v1.5 신설)*: 가사가 검색어에 걸린 곡의 **카드 바로 아래**에 붙는 작은 카드. 걸린 가사 한 줄을 보여주고 검색어 부분을 강조한다.
- 아래는 F-01과 동일 (카드 목록)

### 동작 흐름
1. 사용자가 검색창에 글자를 입력한다.
2. **입력할 때마다 즉시** 목록이 걸러진다. (검색 버튼을 누르거나 엔터를 칠 필요가 없다)
3. 곡의 **제목 또는 가사**에 검색어가 들어 있으면 남기고, 아니면 감춘다.
   - **대소문자를 구분하지 않는다.** (`bandiera`로 `Bandiera`가 찾아진다)
   - 검색어 앞뒤 공백은 무시한다. 공백만 입력한 경우는 검색하지 않은 것과 같다.
4. **가사에 걸린 곡은 카드 아래에 가사 스니펫을 함께 보여준다** *(v1.5 신설)*.
   - 보여주는 것은 검색어가 든 **첫 줄 하나**다. 그 줄 안에 검색어가 여러 번 나오면 **전부** 강조한다.
   - **제목에도 걸렸는지는 따지지 않는다** — 가사에 걸렸으면 보여준다.
   - 제목에만 걸린 곡은 스니펫이 붙지 않는다. (카드 제목이 이미 근거를 보여준다)
5. 걸러진 결과에도 **F-01의 정렬 규칙이 그대로 적용된다** — 분류 순서, 미사용 곡은 맨 아래.
6. 지우기 버튼을 누르면 검색어가 비워지고 전체 목록으로 돌아간다. (스니펫도 함께 사라진다)

> **왜 '검색 버튼'이 없나**: 데이터가 34곡짜리 JSON이라 브라우저 안에서 즉시 걸러집니다 — 네트워크 호출이 없어 기다릴 이유가 없습니다. 버튼을 두면 탭 한 번이 늘 뿐입니다. (곡이 수백 개로 늘거나 서버 검색으로 옮기면 이 결정을 다시 봐야 합니다 — API_SPEC.md 3.1의 `q` 파라미터가 그때를 위한 자리입니다.)

> **왜 '미사용' 곡도 검색되나**: 목록에서 미사용 곡을 아래로 내리는 건 신규팬이 **먼저 만나지** 않게 하려는 것이지, 감추려는 게 아닙니다. 옛 응원가를 일부러 찾는 사람은 제목을 알고 검색합니다 — 이때 안 나오면 고장으로 보입니다.

> **왜 가사 스니펫을 붙이나** *(v1.5)*: v1.4는 가사로 검색해도 **제목만 적힌 카드**를 돌려줬습니다. 팬은 자기가 기억하는 소절이 그 곡에 정말 있는지 확인하려면 **카드를 열어 상세 화면까지 들어갔다 되돌아 나와야** 했습니다 — 결과가 여러 개면 이 왕복을 곡마다 반복해야 합니다. 검색이 "곡을 빨리 찾는" 기능인데 확인 비용이 그만큼 커지면 목적이 깎입니다. 근거가 된 가사 한 줄을 목록에서 바로 보여주면 그 왕복이 사라집니다.

> **왜 '첫 줄 하나'만 보여주나** *(v1.5)*: 응원가 가사는 **같은 소절이 반복되는 구조**입니다(예: '나의 사랑 인천 FC'는 `나의사랑 인천FC`가 8번 반복). 걸린 줄을 전부 보여주면 거의 같은 줄이 카드마다 쌓여 목록이 길어지기만 하고, 정작 "이 곡이 맞나" 판단에는 첫 줄로 충분합니다. 가사 한 줄은 길어야 54자라 한 줄 안에 들어갑니다.

### 상태별 처리
- **정상(검색어 없음)**: 전체 곡을 F-01 규칙대로 표시한다. **스니펫은 붙지 않는다.**
- **정상(검색 결과 있음)**: 걸러진 곡만 표시하고, 가사에 걸린 곡에는 스니펫을 붙인다.
- **검색 결과 없음**: `"○○" 검색 결과가 없습니다` 문구를 보여준다. **검색창은 그대로 둔다** — 검색어를 고쳐야 하므로 없애면 안 된다.
- **곡이 0개(빈 상태)**: 검색창 자체를 표시하지 않는다. (검색할 대상이 없다)

### 완료 조건 (체크리스트)
- [ ] 제목의 일부를 입력하면 해당 곡이 남는다.
- [ ] 가사에만 있는 단어를 입력해도 해당 곡이 찾아진다.
- [ ] 영어 제목을 소문자로 입력해도 찾아진다.
- [ ] 검색 결과에도 정렬 규칙(미사용 맨 아래)이 유지된다.
- [ ] 결과가 없으면 안내 문구가 나오고, 검색창은 화면에 남아 있다.
- [ ] 지우기 버튼을 누르면 전체 목록으로 돌아온다.
- [ ] 곡이 0개면 검색창이 보이지 않는다.
- [ ] **가사에 걸린 곡의 카드 아래에 그 가사 한 줄이 보이고, 검색어 부분이 강조된다.** *(v1.5)*
- [ ] **제목에만 걸린 곡에는 스니펫이 붙지 않는다.** *(v1.5)*
- [ ] **한 줄에 검색어가 두 번 이상 나오면 전부 강조된다.** *(v1.5)*

## F-19. 구단 정보 페이지 빠른 탐색 (플로팅 내비게이션) *(v1.6 신설)*

### 개요
- 길어진 구단 정보 페이지(F-16~F-18)에서 원하는 섹션으로 빠르게 이동할 수 있도록 플로팅 메뉴를 제공합니다.

### 화면 구성 요소
- **모바일 환경**: 화면 상단에 고정(Sticky)되는 가로 스크롤 탭 바. (별도의 클릭 없이 항상 접근 가능)
- **PC/태블릿 환경**: 화면 우측 여백에 고정(Fixed)된 세로 목차 메뉴.
- **메뉴 항목(목차)**: 다가오는 매치, 상대전적, 상대팀 최근 경기, 주요 선수, 부상 및 결장, 최근 경기 결과, K리그1 순위표.

### 동작 흐름
1. 구단 정보 페이지 렌더링 시, 모바일은 상단에 가로 탭 바가, PC는 우측에 세로 메뉴가 항상 노출된다.
2. 스크롤을 내리면 `IntersectionObserver`가 현재 화면에 가장 많이 보이는 섹션을 추적한다.
3. 현재 활성화된 섹션과 일치하는 탭(메뉴 항목)은 인천 블루(`#2E7DFF`) 색상으로 자동 강조된다.
4. 모바일에서 활성 탭이 화면 밖으로 밀려날 경우, 해당 탭이 중앙에 오도록 탭 바가 자동으로 가로 스크롤된다.
5. 사용자가 특정 메뉴 항목을 클릭하면, 화면이 해당 섹션 위치로 부드럽게 스크롤(Smooth Scroll)된다.

### 완료 조건 (체크리스트)
- [ ] 구단 정보 페이지에 진입하면 모바일 상단 탭 바 또는 PC 우측 세로 메뉴가 보인다.
- [ ] 스크롤 시 현재 보고 있는 섹션의 탭이 파란색으로 자동 강조된다.
- [ ] 탭 클릭 시 해당 섹션으로 부드럽게 스크롤 이동한다.
- [ ] 모바일 환경에서 탭 클릭 및 스크롤 시, 선택된 탭이 화면에 보이도록 가로 스크롤이 자동으로 조절된다.

## 공통 데이터 규칙 (참고)

기능들이 공통으로 사용하는 데이터 형식입니다. (PRD 6번과 동일)

```json
{
  "id": 1,
  "title": "응원가 제목",
  "videos": [
    { "type": "official", "label": "공식 영상", "youtubeId": "dQw4w9WgXcQ" },
    { "type": "live",     "label": "현장 영상", "youtubeId": "abcd1234EFG" }
  ],
  "lyrics": "응원가 가사\n(줄바꿈 포함)",
  "category": "팀 응원가",
  "tags": ["대표곡"],
  "isFavorite": false,
  "source": { "name": "인천네이션", "url": "https://incheonation.kr/free/6452" }
}
```

- **`tags`에 '대표곡'** → F-01 카드에 '대표곡' 배지 표시. *(v1.1부터 **정렬에는 관여하지 않습니다** — 판단 기준의 주관성 문제로 '대표곡 우선 정렬'을 제외했습니다. 현재 데이터에 이 태그가 붙은 곡은 없습니다.)*
- **`videos` 목록** → F-04 탭 자동 생성의 기준
- **`lyrics`** → F-02 가사 영역에 줄바꿈 그대로 표시
- **`category`가 '선수 응원가'** → F-01 기본 정렬에서 팀 응원가 다음 묶음
- **`tags`에 '미사용'** → F-01에서 '미사용' 배지 + 항상 맨 아래 묶음. 팀을 떠난 선수의 응원가이며, 구단 공식 선수단 명단을 크롤해 기계가 판정합니다(`scraper/classify_chants.py`, CRAWLER_SPEC.md 3.8)
- **`source`** → F-02 가사 아래에 "가사 출처: ○○"로 표시 (PRD 9장 저작권 — 가사 출처 표기).
  `videos`가 아니라 **`lyrics`의 출처**입니다 — 공식 가사와 커뮤니티 영상이 합쳐진 곡은
  `source`가 공식입니다. (CRAWLER_SPEC.md 3.7.2)
  **선택 필드**이며, 없으면 화면에 아무것도 표기하지 않습니다. *(2026-07-17 기준 34곡 전부
  출처가 있습니다 — 손으로 등록했던 id 1·3도 `reconcile_songs.py`가 크롤 결과와 대조해
  채웠습니다. 필드를 필수로 올리지 않는 이유는 앞으로 손으로 곡을 추가할 수 있어서입니다.)*
