## 1. 개요 (Overview)

### 1.1 배경
- 인천 유나이티드 팬들은 경기장 안팎에서 응원가를 부르지만, 가사와 영상이 한곳에 정리된 곳을 찾기 어렵습니다.
- 신규 팬이나 응원가를 아직 외우지 못한 팬이 **한 화면에서 영상 + 가사를 동시에** 확인할 수 있게 돕는 것이 목적입니다.

### 1.2 목표 (Goals)
- (1단계 핵심) 응원가 목록 및 상세(영상+가사) 정보 제공.
- (2단계 핵심) K리그 실시간 순위, 다가오는 매치, 상대 전적, 주요 선수 및 부상자 정보 제공 (현재 구현 완료 단계).
- **(장기 비전 핵심)** 단순한 정적 정보 제공을 넘어, 클라우드 DB(Supabase)를 도입하여 **과거 인천 유나이티드 팀 기록 및 역대 선수들의 스탯, 히스토리 데이터베이스를 구축**한다.
- **모바일 우선(Mobile-First)** 아키텍처와 빠른 성능을 유지한다.

### 1.3 이번 버전에서 하지 않는 것 (Non-Goals)
> 처음부터 모든 걸 만들려 하면 완성이 늦어집니다. '지금 안 할 것'을 정하는 것도 기획의 핵심입니다.
- 회원가입/로그인 기능
- 사용자가 직접 응원가를 등록하는 기능
- 댓글, 좋아요 같은 커뮤니티 기능

## 3. 핵심 기능 (MVP - 이번에 만들 것)

### F-01. 응원가 목록 화면
- 인천 유나이티드 응원가를 카드(또는 리스트) 형태로 나열합니다.
- 각 항목에는 최소한 **응원가 제목**이 보입니다.
- 항목을 탭(클릭)하면 상세 화면으로 이동합니다.
- **목록 정렬 기준**: **분류 순서**로 배치합니다 — 팀 응원가 → 현역 선수 응원가 → '미사용'(팀을 떠난 선수) 순, 각 묶음 안에서는 가나다순. 분류는 구단 공식 선수단 명단을 크롤해 기계가 판정하며(`category`·`tags`, `scraper/classify_chants.py`), 신규팬이 지금 불리지 않는 곡을 먼저 배우지 않게 합니다. *(v1.9 — 초기 'v1.1 대표곡 우선 배치'는 '무엇이 대표곡인가'의 주관성 문제로 폐기되고 이 객관적 분류 정렬로 대체됐습니다. 상세: FUNCTION.md F-01 v1.3)*

### F-02. 응원가 상세 화면
> **화면 배치 순서 확정 (v1.1)**: 위에서부터 **① 영상 → ② 가사** 순서. 세 페르소나 모두 멜로디/분위기를 먼저 확인하고 싶어 하기 때문입니다.

- 화면 상단에 **영상 선택 탭 [공식 영상] [현장 영상]** 을 둡니다. *(F-04 참고)*
- 선택된 **유튜브 영상**이 재생 가능한 형태로 표시됩니다.
- 영상 **아래**에 **가사**가 표시됩니다.
- 목록으로 돌아가는 '뒤로 가기' 동선이 있습니다.

### F-04. 영상 종류 선택 (공식 / 현장) *(v1.1 신설)*
> **왜 '나란히'가 아니라 '선택'인가**: 모바일 화면은 폭이 좁아 영상 2개를 동시에 두면 너무 작아지고, 둘 다 불러오면 로딩이 느려집니다. 그래서 **한 번에 하나만** 보여주는 탭 방식을 채택합니다.

- 상세 화면 상단에 **[공식 영상] [현장 영상]** 두 개의 탭을 제공합니다.
- **기본 선택값은 '공식 영상'** 입니다. (핵심 타겟인 신규팬의 가사 학습 목적에 가장 적합)
- 탭을 누르면 해당 종류의 영상으로 전환됩니다.
- **예외 처리**: 어떤 곡에 한 종류의 영상만 있는 경우(예: 현장 영상 없음), 없는 탭은 숨기거나 비활성화합니다.

### F-03. 모바일 친화 레이아웃
- 세로 화면(스마트폰) 기준으로 콘텐츠가 자연스럽게 배치됩니다.
- 유튜브 영상이 화면 너비에 맞춰 크기가 조절됩니다(반응형).
- 글씨 크기와 버튼(탭 포함)이 손가락으로 누르기 편한 크기입니다.

## 6. 데이터 구조 (Data Model) ⭐가장 중요한 뼈대

> 응원가 하나가 담고 있어야 할 '정보 서랍'을 미리 정의합니다.
> **v1.1 변경**: 영상 주소 칸을 1개(`youtubeId`)에서, 여러 영상을 담는 **목록(`videos`)** 형태로 확장했습니다.
> 이렇게 하면 나중에 '연습 영상', '아카펠라 영상' 같은 종류가 늘어도 구조를 바꾸지 않고 항목만 추가하면 됩니다.

응원가 1곡(하나의 데이터) 예시:

```json
{
  "id": 1,
  "title": "응원가 제목",

  "videos": [
    { "type": "official", "label": "공식 영상", "youtubeId": "dQw4w9WgXcQ" },
    { "type": "live",     "label": "현장 영상", "youtubeId": "abcd1234EFG" }
  ],

  "lyrics": "응원가 가사 내용\n(줄바꿈 포함)",

  "category": "팀 응원가",     // 확장용
  "tags": ["대표곡"],          // 확장용 (목록 정렬·필터에 사용)
  "isFavorite": false,         // 확장용 (즐겨찾기)

  // 가사를 어디서 가져왔는지 (9장 저작권 — 가사 출처 표기). 출처를 모르는 곡은 이 칸이 없습니다.
  "source": { "name": "인천네이션", "url": "https://incheonation.kr/free/6452" }
}
```

- `videos`는 **여러 영상을 담는 목록**입니다. 각 영상은 종류(`type`), 화면에 보일 이름(`label`), 유튜브 ID(`youtubeId`)를 가집니다.
- 상세 화면의 **[공식 영상]/[현장 영상] 탭은 이 `videos` 목록을 그대로 읽어서** 자동으로 만들어집니다. (영상이 1개뿐이면 탭도 1개만 생김)
- `source`는 **가사의 출처**입니다 *(2026-07-17 신설)*. 영상은 유튜브에서 그대로 비춰 보여주니 출처가 자명하지만, 가사는 어디서 옮겨 적었는지 밝히는 것이 9장의 저작권 방침입니다. 상세 화면 가사 아래에 "가사 출처: ○○"로 작게 표시되며, 누르면 원문으로 갑니다.
- 초기에는 별도 DB 없이 **하나의 파일(예: `songs.json`)** 에 저장하는 것으로 시작했습니다.
- 하지만 과거 팀/선수 데이터의 지속적인 아카이빙 비전에 따라, **JSON 데이터를 Supabase 등 클라우드 DB로 마이그레이션(이관)하는 작업**이 예정되어 있습니다.

## 7. 기술 스택 제안 (Tech Stack)

| 영역 | 제안 | 이유 (초심자 관점) |
|------|------|------|
| 프레임워크 | Next.js + React | 화면을 '부품(컴포넌트)' 단위로 나눠 만들기 쉬움 |

- **반응형**: 스마트폰(우선), 태블릿, PC에서 깨지지 않게 보입니다.
- **접근성**: 글자 대비를 충분히 확보해 야외에서도 잘 보이게 합니다.

## 9. 유의사항 (Considerations)

- **저작권**: 응원가 가사와 유튜브 영상은 저작권이 있을 수 있습니다.
  - 유튜브 영상은 직접 재업로드하지 말고 **공식 임베드(iframe) 방식**으로 불러오는 것이 안전합니다. (공식/현장 영상 모두 동일)
  - 가사는 출처를 함께 표기하거나, 공식/공개된 자료를 사용하는 방향을 권장합니다.
  - **(2026-07-17 구현 완료)** 위 두 가지를 모두 적용했습니다: 가사는 **공식 홈페이지(구단 게시)** 와 **인천네이션(팬 커뮤니티 공개글)** 이라는 공개 자료에서 수집하고, 곡별로 `source`(6장)를 저장해 상세 화면에 "가사 출처: ○○"를 원문 링크와 함께 표시합니다. 수집 방식·소스 선정 근거는 CRAWLER_SPEC.md 3.7절 참고. 수집 대상 사이트의 `robots.txt`를 준수하며, 봇 차단을 우회해야 하는 소스(나무위키)는 채택하지 않았습니다.
- **데이터 출처 표기 범위** *(2026-07-17 신설)*: 저작권과 별개로, **어디서 가져온 데이터인지 사용자가 알 수 있어야** 합니다. 응원가 가사뿐 아니라 구단 정보(F-16~F-18)도 화면 하단에 수집처를 작게 표기합니다 — 순위표·경기·상대 폼·상대전적 **전부 다음 스포츠** 하나입니다(F-18의 상대전적이 API-Football에서 넘어오며 통일됐습니다). 표기 스타일은 BASE.md 4장의 '미세 정보'(12px) 역할을 씁니다.
- **데이터 준비**: 개발 전에 곡별로 제목 / **공식·현장 영상 ID** / 가사를 정리한 목록을 미리 만들어두면 개발이 훨씬 수월합니다.
