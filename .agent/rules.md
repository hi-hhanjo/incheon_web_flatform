- **문서 버전**: v1.3 · **작성일**: 2026-07-11 (v1.3 업데이트: 2026-07-17)

- **목적**: 이 프로젝트에서 코드를 작성·수정할 때(사람과 AI 모두) 반드시 지켜야 할 규칙을 정의합니다.
- **어떻게 적용되나** *(v1.2)*: 루트 [`CLAUDE.md`](../CLAUDE.md)가 이 파일을 `@.agent/rules.md`로 import합니다. *(v1.1까지는 "AI 코딩 도구는 이 파일을 먼저 읽습니다"라고 적혀 있었지만 **실제로 이 파일을 읽는 도구는 없었습니다** — 선언만 있고 연결이 없었습니다. 2026-07-17 import를 걸어 실제로 적용되게 했습니다.)*

## 1. 문서를 진실의 근거로 삼는다 (Single Source of Truth)

- 아래 문서들이 **정답의 기준**이다. 임의로 상상해서 만들지 않는다.
  - `docs/bible/PRD.md` — 무엇을/왜 만드는가 (현재 범위·Non-Goals·저작권 방침의 기준)
  - `docs/bible/FUNCTION.md` — 각 기능이 어떻게 동작하는가
  - `docs/bible/TECHSTACK.md` — 어떤 기술로 만드는가
  - `docs/design/BASE.md` — 색·글꼴·간격 토큰
  - `docs/design/README.md` → `docs/design/components/*.md` — 각 부품의 상세 명세
  - `docs/data/API_SPEC.md` · `ERD_SPEC.md` · `CLUB_ERD_SPEC.md` — 데이터 모델·API 계약
  - `docs/data/CRAWLER_SPEC.md` — 데이터를 어디서 어떻게 긁어오는가
  > *(v1.2 — 이전엔 `PRD.md`·`DESIGN.md`처럼 루트 기준 이름으로 적혀 있었으나 실제 파일은 `docs/` 아래로 옮겨져 있어 경로를 맞췄습니다. `DESIGN.md`의 현재 이름은 `BASE.md`, `기능명세서.md`는 `FUNCTION.md`입니다.)*
- 문서에 없거나, 문서끼리 충돌하거나, 애매한 부분이 있으면 → **임의로 정하지 말고 먼저 질문한다.**
- 문서와 다르게 구현해야 할 이유가 생기면 → **먼저 문서를 고치고**, 그다음 코드를 작성한다. (코드가 문서를 앞서가지 않는다)

---

## 2. 범위를 지킨다 (Scope Discipline)

> 원칙만 담고, '지금의 범위'는 문서를 가리킨다. (v1.1에서 이렇게 정리)

- **요청받지 않은, 범위 밖 기능을 미리 만들지 않는다.** ("있으면 좋을 것 같아서" 추가 금지)
- **현재 구현 범위와 '만들지 않는 기능(Non-Goals)'의 기준은 항상 `PRD.md`와 `FUNCTION.md`다.** 범위가 바뀌면 그 문서를 고치고, 이 파일은 그대로 둔다.
- 명세에 없는 기술 요소(별도 백엔드 서버, 데이터베이스, 상태관리 라이브러리 등)를 **임의로 추가하지 않는다.** 추가가 필요하면 3번 절차(이유 설명 → 확인)를 따른다.
- 확장을 고려해 설계된 데이터 구조(`videos`, `tags` 등)는 지금 다 쓰지 않더라도 **그대로 유지한다.**

---

## 3. 기술 스택을 고정한다 (TECHSTACK.md 준수)

- 사용 기술은 `TECHSTACK.md`에 정의된 것만 사용한다. (Next.js + React, Tailwind CSS, JSON 파일, YouTube 임베드, Vercel)
- 새 라이브러리·패키지를 추가하려면 → **먼저 이유를 설명하고 확인을 받는다.** (무분별한 의존성 추가 금지)
- 폴더 구조는 `TECHSTACK.md 4장`의 구조를 따른다.

---

## 4. 디자인 규칙을 벗어나지 않는다 (BASE.md 준수)

- 색상·폰트·간격·모서리 값은 **BASE.md에 정의된 값만** 사용한다. 임의의 값을 하드코딩하지 않는다. (구체적 색상값의 출처는 BASE.md다)
- 색·간격 등은 가능하면 **공통 토큰(Tailwind 설정 등)으로 관리**하고, 값을 여기저기 흩어 쓰지 않는다.
- **인천 블루는 기능(재생·선택·주요 버튼·대표곡 배지)에만** 쓴다. 장식용으로 남발하지 않는다.
- 가사는 반드시 크게·넉넉한 행간으로 표시한다. (BASE.md 3장 기준)

---

## 5. 컴포넌트를 재사용한다 (components/*.md 준수)

- 화면을 만들 때는 `components/` 폴더의 부품을 **먼저 재사용**한다. 같은 부품을 중복해서 새로 만들지 않는다.
- 각 부품은 해당 명세 파일(예: `song-card.md`)의 입력(Props)·상태·스타일을 그대로 따른다.
- 부품 사이 연결 관계(`components/README.md 3장`)를 지킨다.
  - 예: 영상은 `VideoTabs`에서 고른 것을 `YoutubePlayer`가 표시한다.

---

## 6. 데이터 구조를 지킨다 (Data Model)

- `songs.json`의 형식은 PRD 6장 / 기능명세서 '공통 데이터 규칙'을 따른다.
- 필드 이름(`id`, `title`, `videos`, `lyrics`, `tags` 등)을 **임의로 바꾸지 않는다.**
- 영상은 항상 `videos` **목록(배열)** 으로 다룬다. (단일 `youtubeId`로 되돌리지 않는다)
- 화면은 데이터를 읽어 **자동으로** 그린다. (탭 개수, 대표곡 배지 등을 코드에 하드코딩하지 않는다)

---

## 7. 예외·빈 상태를 반드시 처리한다

- 모든 화면은 **정상 / 로딩 / 빈 상태 / 에러**를 처리한다. 화면이 비어 보이게 두지 않는다. (`feedback.md` 사용)
  - 곡 없음 → "등록된 응원가가 없습니다"
  - 영상 없음 → "영상 준비 중입니다"
  - 가사 없음 → "가사 준비 중입니다"
  - 없는 곡 접근 → "해당 응원가를 찾을 수 없습니다" + 목록 이동 버튼
- 기능명세서의 **완료 조건 체크리스트**를 만족해야 그 기능이 '완료'다.

---

## 8. 모바일 우선·접근성을 지킨다 (F-03)

- **모바일 세로 화면**을 기준으로 만든다. 어떤 화면에서도 **가로 스크롤이 생기면 안 된다.**
- 누를 수 있는 요소(카드·탭·버튼)는 **최소 44px** 터치 영역을 확보한다.
- 유튜브 영상은 **16:9 비율**을 유지하며 화면 폭에 맞춘다.
- 야외 가독성을 위해 글자 대비를 충분히 유지한다.

---

## 9. 저작권을 지킨다 (PRD 9장)

- 유튜브 영상은 **임베드(iframe)로만** 불러온다. 영상 파일을 직접 복사·재업로드하지 않는다.
- 가사는 **출처가 확인된 자료**만 사용하고, 필요 시 출처를 함께 표기한다.

---

## 10. 작게, 확인하며 진행한다 (Incremental & Safe)

- 한 번에 하나의 화면/부품 단위로 작업한다. 프로젝트 전체를 한꺼번에 갈아엎지 않는다.
- 큰 변경(구조 변경, 파일 대량 수정, 의존성 추가) 전에는 **무엇을·왜 바꾸는지 먼저 설명하고 확인**을 받는다.
- 기존에 잘 동작하던 부분을 이유 없이 바꾸지 않는다.

---

## 11. 언어·표기 규칙

- **화면에 보이는 글자(UI 문구)와 주석·문서는 한국어**로 작성한다.
- **코드 식별자(변수·함수·파일명)는 영어**로 작성한다. (예: `SongCard`, `videos`)
- 문서 파일의 변경 이력(Changelog)은 변경이 생길 때마다 갱신한다.

---

## 12. AI 코딩 도구를 위한 행동 규칙

- 확실하지 않으면 **추측하지 말고 질문한다.**
- 규칙이나 명세를 **말없이 벗어나지 않는다.** 벗어나야 한다면 이유를 먼저 밝힌다.
- 어떤 결정을 내릴 때, 근거가 된 문서를 짧게 언급한다. (예: "BASE.md 4장에 따라…")
- 비전공자도 이해할 수 있도록, 복잡한 코드에는 **간단한 한국어 주석**을 단다.
